1. Executive Gap Analysis
The current PromptVault build is production-ready in its core commerce loop: users can register, browse, purchase, and download prompt packs, and administrators can run the automation pipeline and approve content. However, when evaluated against the standard of a truly advanced, industry-level SaaS marketplace - comparable to Gumroad, Notion's template gallery, or Envato Market - a significant layer of features is missing that separates a functional MVP from a compelling, sticky, and revenue-maximising platform.
This document is the complete PRD for everything that needs to be built next. It is organised by domain, with each section covering the full screen inventory, database requirements, API endpoints, UI/UX specification, and business logic needed. The goal is to provide a build-ready specification that requires zero ambiguity to implement.

2. User Profile System (Complete)
The current system has only an account settings page with basic fields. A complete user profile system is one of the highest-leverage missing features - it creates identity, drives social engagement, surfaces purchase history publicly (with privacy controls), and is the foundation for the eventual multi-seller marketplace. Every professional marketplace (Gumroad, Creative Market, Envato) centres the user experience around rich profiles.
2.1 Public Profile Page  [SCR-033]
Route: /u/:username. This is the most important missing screen. It is the face of a user or creator on the platform and must be designed with the same premium quality as a portfolio site.
Layout & Components
• Profile Hero Section: Full-width cover banner image (user-uploadable, 1500×400px). Overlaid: circular avatar (160px, bordered with primary gradient ring), display name in text-5xl bold, username in text-xl muted, join date, location (optional), and website URL (verified link icon). Follow button (for logged-in visitors) and Edit Profile button (for the owner). Share profile dropdown: copy link, Twitter/X, LinkedIn.
• Stats Bar: A horizontal row of four stat chips - Packs Published, Total Prompts, Total Downloads (all-time), and Average Rating - displayed with large monospace numbers and animated count-up on scroll-into-view. These are public and always visible.
• Profile Bio Section: Up to 500 characters of rich text (bold, italic, links only - no heading tags). Renders as HTML. Edit inline for the profile owner. Truncated at 3 lines with "Read more" expand on public view.
• Specialties / Skills Tags: A row of up to 10 user-defined tags (e.g. "ChatGPT", "Marketing", "Midjourney") rendered as styled pill badges in the user's chosen accent colour. Clickable - clicking a tag navigates to search filtered by that tag.
• Social Links Row: Icon links for Twitter/X, LinkedIn, GitHub, YouTube, personal website. Each verified with a checkmark if the user has connected via OAuth. Greyed out and hidden if not set.
• Published Packs Grid: 3-column responsive grid of all the user's published packs, identical to the marketplace pack cards but filtered to this creator. Sorted by most recent by default. "Sort by: Newest / Most Downloaded / Top Rated" selector. Pagination at 12 per page.
• Reviews Received Section: Aggregate star display (large), histogram of 1-5 stars, and a paginated list of reviews left on the user's packs. Each review card shows the pack it was for.
• Activity Timeline (optional, user-configurable): A condensed list of recent public activity - pack published, collection created, milestone reached. User can toggle this section off in privacy settings.

Verified status is stripped automatically if rating drops below 3.5 for 30 consecutive days.


3. Social & Community Features
Social mechanics are the primary driver of organic growth and retention in any marketplace. They reduce dependency on paid acquisition by creating network effects - a user who follows a creator has a reason to return daily. The features in this section transform PromptVault from a transactional shop into a living community.
3.1 Collections (User-Created Playlists)  [SCR-041]
Route: /collections/:id and /dashboard/collections. Collections allow any user to organise prompts and packs into curated lists, similar to Spotify playlists or Pinterest boards. They are a critical engagement driver because they are shareable, discoverable, and give users a reason to return and update their content.
• A collection has: title (100 chars), description (500 chars), cover image (auto-generated from first item or user-uploaded), visibility (public/private/followers), and an ordered list of items (prompts or packs).
• Users can create unlimited collections from their dashboard. A "Save to Collection" button appears on every pack detail page and prompt card - clicking opens a modal showing the user's existing collections with checkboxes and a "New Collection" option at the top.
• Public collections are discoverable via search and appear on the creator's public profile. The platform's admin can feature collections on the homepage.
• Collection detail page: cover image, title, creator info, item count, last updated, Follow Collection button (subscribers get notified when items are added), and the ordered grid of items. Drag-to-reorder for the owner. Share button.
• Collections widget on the homepage: "Staff Picks" and "Trending Collections" sections showing 4 collection cards each.
3.2 Prompt Comments & Discussion  [SCR-042]
Each pack detail page needs a Comments tab alongside Reviews. Comments are different from reviews: they are not rated, they can be replies, and they are for discussion ("Has anyone tried this with Claude 3.5?"). This is the #1 feature request in comparable marketplaces.
• Threaded comments: top-level comments and one level of replies only (no infinite nesting). This keeps the UI clean.
• Markdown support: bold, italic, inline code, links. Code blocks rendered with syntax highlighting (Prism.js) for sharing prompt variations.
• Upvote system: each comment can be upvoted (not downvoted) to surface most helpful comments. Top comment pinned with gold ring.
• Comment author badges: "Pack Owner" badge if the creator comments on their own pack, "Verified Buyer" if they have purchased.
• Moderation: comments are soft-deleted (not hard). Flag button on each comment. Admins see flagged comments in the moderation queue. Auto-hide comments with 3+ flags pending review.
• Pagination: load 10 top-level comments, "Load more" button. Replies collapsed by default, "View X replies" expander.
3.3 Like / Appreciate System  [SCR-043 - Component]
A lightweight appreciation mechanic (distinct from reviews) for quick positive feedback. Each pack has an "Appreciate" button (heart icon with count) separate from the star rating. Any logged-in user (including non-buyers) can appreciate a pack. The count is public. Appreciation is used in the trending algorithm and the creator's profile stats. Unlike reviews, appreciations are anonymous to other users (only the total count is shown). Creators can see who appreciated from their analytics dashboard.
3.4 Sharing & Referral Deep Links  [SCR-044 - Component]
Every pack, profile, and collection needs a professional sharing experience. The Share button (already present) needs to be upgraded from a simple copy-link into a full sharing modal. The modal shows: a pre-rendered social card preview (OG image with pack thumbnail, title, and PromptVault branding), platform-specific share buttons (Twitter/X with pre-filled tweet, LinkedIn, WhatsApp, Telegram, Reddit, email mailto), a copy-link button with animation, and an "Earn commission" prompt for logged-in users linking to the affiliate program. All share links include UTM parameters for analytics attribution.
3.5 User-to-User Messaging  [SCR-045]
Route: /dashboard/messages. A lightweight direct messaging system for creator-to-buyer communication. This is needed for custom pack requests, support, and community building. Not a general chat - messages can only be initiated by a buyer to a creator whose pack they have purchased, or by any verified creator to any user. This scoping prevents spam.
• Inbox view: list of conversations, each showing the other person's avatar, name, message preview, timestamp, and unread count badge.
• Conversation view: message thread with timestamped bubbles, text-only (no file attachments in V1), 2000-char limit per message. Markdown: inline code and links only.
• New message: accessible via a "Message Creator" button on a pack detail page (only if the user has purchased) or from a user's public profile.
• Notifications: new messages trigger the notification badge in the nav. An email notification is sent after 30 minutes of unread messages.
• Admin oversight: admins can view any conversation thread from the user management panel for moderation purposes. Users are informed of this in the Privacy Policy.
3.6 Community Prompts (Free Tier)  [SCR-046]
Route: /community. A free, publicly visible stream of prompts submitted by the community - think Product Hunt for prompts. Any registered user can submit a single prompt for free. Submissions are voted on (upvote only) by the community. The top-voted prompts each week are highlighted on the homepage. Creators can optionally convert a highly-voted community prompt into a full paid pack. This drives organic SEO content and user-generated engagement. Each community prompt has: the prompt text, the submitter's profile, AI tool tag, upvote count, comment count, submit date, and a "Save to Collection" button. Admins can feature, remove, or convert community prompts.


4. Advanced Search & Discovery
Search is the primary discovery mechanism for any content marketplace. The current ILIKE fallback is adequate for MVP but falls short on relevance, typo tolerance, and semantic understanding. This section specifies the full search upgrade needed to make PromptVault a world-class discovery experience.
4.1 AI-Powered Semantic Search  [Backend Upgrade]
The current search uses PostgreSQL ILIKE which only matches exact substrings. A user searching "grow my business with AI" will not find a pack called "ChatGPT Marketing Prompts for Revenue Growth." Semantic search bridges this gap by understanding intent, not just keywords.
• Implementation: generate text embeddings for every pack's title + description + tags using the OpenAI text-embedding-3-small model (1536 dimensions, low cost). Store vectors in pgvector extension on PostgreSQL.
• At search time: embed the user's query, then run a cosine similarity search against the pack embeddings. Combine with a BM25 keyword score (Meilisearch) using a 70/30 weighted blend. The result is a hybrid search that handles both keyword precision and semantic intent.
• Re-embed packs automatically when title, description, or tags change (BullMQ job triggered on pack update).
• New DB table: pack_embeddings (packId UUID PK, embedding VECTOR(1536), model VARCHAR, createdAt, updatedAt). Requires pgvector extension.
4.2 Advanced Search Page Upgrade  [SCR-047]
The current search page needs significant expansion. The upgraded version has three major new capabilities: semantic understanding (via embeddings), advanced filter combinations, and saved searches.
• Search bar: full-width, autofocused, with an AI mode toggle (sparkle icon) that switches between keyword and semantic search modes. AI mode shows a subtle "✦ AI-powered" indicator.
• Advanced Filters Panel: expanded from the current basic filters to include - AI tool (multi-select: ChatGPT, Claude, Midjourney, Gemini, Cursor, Notion AI, DALL-E, Runway, Sora, and more), use case (multi-select: marketing, coding, writing, design, etc.), prompt count range (slider: 10-1000+), language (for non-English packs), content rating, date range (published within: 24h/7d/30d/90d/all time), minimum rating slider, and price range.
• Saved Searches: a "Save this search" button stores the current query + filters as a named saved search in the user's account. Saved searches appear in a dropdown from the search bar on focus. Users can optionally subscribe to a saved search (daily email digest of new matching packs).
• Search suggestions: as-you-type dropdown with: top matching pack names, popular related tags, category suggestions ("Browse all Marketing packs →"), and recent searches. Keyboard-navigable.
• No-results experience: when zero results are found, show - "Did you mean X?" typo correction, related categories with pack counts, trending packs from the most relevant category, and a "Request a Pack" CTA that lets users submit a prompt pack idea (goes to admin queue).
• Search analytics: every search logs query, result count, clicked result (position), and whether a purchase followed within the session. These signals feed relevance ranking improvements.
4.3 Personalised Recommendations  [SCR-048 - Component]
A "Recommended For You" section shown on the homepage (for logged-in users), the explore page, and after each purchase. Recommendations are computed by a nightly BullMQ job that runs a collaborative filtering algorithm: users who purchased the same packs as you also bought these other packs. Fall back to category-based recommendations for new users with fewer than 3 purchases. The recommendation widget shows 4 pack cards in a horizontal scroll with a "Why recommended" tooltip on hover: "Because you bought X" or "Popular in Marketing."
4.4 Trending & Charts  [SCR-049]
Route: /trending (expanded from current). The current trending page is a simple ranked list. The upgraded version is a full charts experience similar to the App Store or Bandcamp. Sections: "#1 This Week" hero card (featured prominently), "Rising" chart (packs new in the last 7 days with most growth momentum, calculated as downloads[day 1-7] vs downloads[day 8-14] ratio), "All-Time Bestsellers" chart, "New Releases" (published in the last 7 days), and "Underrated Gems" (high rating, low download count - surfacing quality packs that deserve more visibility). Each section is a horizontally scrollable row with rank badges. The ranking algorithms run via BullMQ nightly job, results cached in Redis.

6. Prompt Interaction Features
Individual prompt interactions are almost entirely absent from the current build. Users who have purchased a pack need rich tools to work with their prompts - copy, customise, organise, and rate them at the individual level, not just the pack level.
6.1 Prompt Viewer (Full Screen)  [SCR-052]
Route: /dashboard/packs/:packId/prompts. A full-screen prompt browser for purchased packs. This replaces the basic "Download Center" for users who want to browse and use prompts in-browser without downloading.
• Layout: two-panel. Left: list of all prompts in the pack with title, use-case badge, and a coloured dot indicating whether it has been used (user-tracked). Right: the selected prompt in full detail.
• Prompt Detail Panel: large prompt text in a styled code block (Fira Code font, dark background), variable highlighter (placeholder variables like {topic} shown in a contrasting colour with editable inline inputs), copy-to-clipboard button with a "Copied!" animation, open-in-ChatGPT button (deep link to chat.openai.com with the prompt pre-filled), open-in-Claude button (deep link to claude.ai), personal rating (1-5 stars, private to the user, used for "My Favourites" filtering), and a "Add to Collection" button.
• "Mark as Used" toggle on each prompt: tracks usage locally and syncs to DB. Shows a "Used" count in the left panel list. This data is shown to the creator in analytics ("Prompt #12 has the highest usage rate").
• Search within pack: a small search bar in the left panel filters prompts by title or body text in real-time (client-side, no API call needed since all prompts are loaded).
• Variable editor: when a prompt contains {variable} placeholders, a sidebar panel shows each variable as a labelled input field. As the user types, the main prompt text updates in real-time with their values filled in. The "Copy" button then copies the completed, filled-in version.
6.2 Prompt Favourites & Personal Library  [SCR-053]
Route: /dashboard/library. A cross-pack personal library of individual prompts the user has bookmarked. This is distinct from the pack wishlist - it operates at the individual prompt level, allowing users to build a personalised "best of" collection from across all their purchased packs.
• Grid/list toggle. Filter by: AI tool, use case, pack it came from, personal rating, recently used. Sort by: date bookmarked, personal rating, use-case.
• Each prompt card shows: prompt title, pack name (linked), AI tool badge, use-case tag, personal star rating, copy button, and remove-from-library icon.
• Export library: button to download all favourited prompts as a personal PDF or JSON file. This is a premium feature (available to Pro subscribers and buyers only).
• DB: new table prompt_bookmarks (userId, promptId, personalRating SMALLINT, usedCount INT, lastUsedAt, createdAt) - composite PK on userId + promptId.
6.3 Prompt Customisation & Templates  [SCR-054 - Component]
Variable-injection is the most powerful usability upgrade for prompts. Many professional prompts use placeholder variables (e.g., "Write a {tone} email about {topic} for {audience}"). The platform should make these first-class features rather than text in a code block.
• When a creator builds a prompt (in the admin editor), they can mark variables using a {variable_name} syntax. The editor shows a "Variables Detected" panel listing each variable with a field for: display label, placeholder text, and type (text / select / number).
• When a buyer views the prompt, the variable panel auto-renders these inputs. The prompt preview updates live. This is saved to the DB as variables JSONB on the Prompt model (already in schema).
• Creators can publish "template packs" where variable injection is the core value prop - the prompts are frameworks that buyers fill in for their specific situation.


7. Creator / Seller Expansion
PromptVault is currently a single-seller platform (the platform itself generates and sells all packs). Expanding to a multi-seller marketplace is the highest-leverage growth move - it turns the platform into a two-sided marketplace with network effects, reduces the content generation burden, and opens a commission-based revenue stream.
7.1 Creator Application & Onboarding  [SCR-055]
Route: /become-a-creator. A dedicated landing page explaining the creator programme with benefits, commission rates, and success stories. A prominent "Apply to Sell" CTA leads to a multi-step onboarding form.
• Step 1 - Eligibility Check: auto-check if the user meets minimum criteria (verified email, account age, no violations). If not, show a friendly "Not quite yet" screen with the specific criteria and progress toward each.
• Step 2 - Creator Profile: display name, bio (creator-specific), specialties, payout info introduction.
• Step 3 - Payout Setup: Stripe Connect Express onboarding (redirect to Stripe hosted flow). Platform handles payouts via Stripe Connect. Creator must complete Stripe's KYC (identity verification) before they can receive payouts. Minimum payout threshold: $25.
• Step 4 - Agreement: display Creator Agreement (terms, commission rates, content policy). Checkbox consent with timestamp stored.
• Step 5 - First Pack Prompt: optional guided prompt to create their first pack immediately - "Create your first pack to get approved faster."
• Application goes to admin review queue. Admin can approve, request more info, or reject with a reason. Email sent either way.
7.2 Creator Dashboard  [SCR-056]
Route: /creator/dashboard. A completely separate dashboard section for approved creators (in addition to their regular user dashboard). Accessible via a "Creator Studio" link in the nav dropdown.
Creator Dashboard Sections
• Revenue Overview: total earnings (lifetime, this month, last month), pending payout, next payout date. Bar chart of daily earnings for the last 30 days.
• Pack Performance Table: all published packs with columns - title, status, views, purchases, conversion rate, revenue this month, total revenue, average rating, last updated. Sortable. Click row to open pack analytics.
• New Pack button: leads to the creator pack creation flow (not the admin automation flow).
• Payout History: table of all payouts received, with date, amount, Stripe transfer ID, and status.
• Audience Insights: total buyers, returning buyers, top countries, most active time of day. Privacy-compliant (no individual buyer data exposed).
7.3 Creator Pack Builder  [SCR-057]
Route: /creator/packs/new and /creator/packs/:id/edit. A rich, guided pack creation interface for community creators (distinct from the admin automation pipeline). Designed to be approachable for non-technical creators while producing high-quality structured packs.
• Step 1 - Pack Details: title, description (rich text editor with markdown), short description, category (searchable select), AI tool targets (multi-select), and tags.
• Step 2 - Add Prompts: an inline prompt editor with fields for title, body (the prompt), use case, AI tool, and variables. Bulk import option: paste a text file (one prompt per line with a delimiter) or upload a CSV. AI assist button: "Generate 10 more prompts similar to this one" - calls the AI API using the existing prompt as context and adds the results for the creator to review and edit.
• Step 3 - Pricing: price input with suggestions from the platform ("Based on similar packs, we suggest $14.99"). Free toggle. Compare price (strike-through) for promotional pricing.
• Step 4 - Media: thumbnail upload with AI-generate option (calls the image generation service). Preview image upload. Pack preview PDF is auto-generated after submission.
• Step 5 - Review & Submit: full preview of the pack as it will appear on the marketplace. Submit for Review button sends to admin moderation queue with creator_id attribution.

8. Advanced Commerce Features
The current commerce system handles single-item purchases cleanly. However, several high-value commerce patterns are missing that would meaningfully increase average order value and repeat purchase rate.
8.1 Bundle Packs  [SCR-060]
Bundles allow multiple packs to be sold together at a discount. They are one of the highest-leverage AOV-increasing features in any digital marketplace. A bundle is a special pack type that references 3-10 individual packs. The bundle page shows all included packs with their individual prices, the bundle price, and a large "You Save X%" badge.
• Bundle creation (admin and creators): select packs to include, set bundle price (must be lower than sum of individual prices), set bundle title and description. Thumbnail is auto-generated as a collage of the included pack thumbnails.
• Bundle purchase creates OrderItems for each included pack individually - the buyer gets access to each pack separately in their download center.
• New field on PromptPack: packType ENUM (single, bundle), bundleItems (relation to PromptPackBundleItem table with packId and sortOrder).
• Bundle card in marketplace: shows "X packs included" badge, lists included pack names below the description, and shows the savings prominently.
8.2 Pack Gifting  [SCR-061]
Allow users to purchase any pack as a gift for another user. This opens up a significant gifting use case (team leads gifting packs to their team, gift cards for AI enthusiasts) and is a meaningful driver of new user acquisition.
• On any pack detail page, a "Give as Gift" button (below the main CTA) opens a gift modal: recipient email field, optional personal message (200 chars), option to delay delivery (schedule for a future date), and gift wrapping visual (decorative, no functional difference).
• If the recipient already has an account, the pack is added to their library immediately. If not, a gift redemption email is sent with a unique redemption link that creates an account and adds the pack automatically.
• Gift redemption page [SCR-062]: a special landing page (/gift/:token) with a celebratory design - unwrap animation, pack preview, and "Create your account to claim your gift" CTA.
• DB: new table gift_orders (id, orderId, recipientEmail, senderMessage, scheduledFor, redeemedAt, redemptionToken UUID).
• The gifter's order receipt says "Gift to [recipient email]". The gifter does not see the recipient's personal library.
8.3 Cart & Multi-Item Checkout  [SCR-063]
The current system is a single-item "Buy Now" flow. A proper cart allows users to add multiple packs and check out in one transaction, which increases AOV and reduces checkout abandonment. The cart is a persistent, server-side cart (stored in the DB as a cart table, not localStorage) so it survives page refreshes and device switches.
• Cart icon in the navbar with item count badge. Clicking opens a slide-over cart drawer (not a separate page) showing: pack thumbnails, names, prices, remove buttons, subtotal, coupon code field, discount display, and "Checkout" button.
• Cart drawer also shows: "You might also like" - 2 recommended packs that the user does not already own in the same categories as cart items.
• Packs already owned are visually distinguished (greyed out, "Already owned" badge) if somehow added to cart again - clicking "Checkout" auto-removes them with a toast.
• DB: new tables cart (id, userId UNIQUE FK, updatedAt) and cart_items (cartId FK, packId FK, addedAt - composite PK).
• Cart persists for 30 days. Abandoned cart email: if a user has items in their cart and has not visited for 24 hours, send a reminder email (one time only per cart session). This alone typically recovers 5-15% of abandoned carts.
8.4 Pack Preview / Try Before You Buy  [SCR-064 - Component]
Currently, unregistered users see 5 blurred prompts. This needs upgrading into a more intentional "Try" experience. A "Try 3 Prompts Free" button on any pack detail page allows any visitor (including non-registered) to unlock 3 specific (not random - the creator/admin selects which 3 are most impressive) prompts in full, with copy functionality. After using the free trial, a modal appears: "You've used your 3 free previews. Get all {N} prompts for ${price}" with a countdown timer creating urgency. This dramatically increases conversion from browsing to purchase.
8.5 Dynamic Pricing & Flash Sales  [SCR-065 - Admin Feature]
A flash sale system allowing time-limited price drops. Admin (and eventually creators) can set a sale price and duration on any pack. The pack card shows a countdown timer badge ("Sale ends in 2h 14m"), a red "SALE" badge, and the strike-through original price. Sales are managed via a new SaleEvent table (packId, salePrice, startAt, endAt, createdBy) with a BullMQ job that automatically restores the original price at the endAt time. The homepage has a "Flash Sales" section showing currently active sales with countdown timers.

12. Content Moderation & Trust
As PromptVault opens to community creators, content moderation becomes a critical operational concern. The current basic flagging system is insufficient for a multi-creator marketplace. This section specifies a complete, scalable trust and safety system.
12.1 Automated Content Pre-Screening  [Backend]
Before any AI-generated or creator-submitted pack enters the human moderation queue, it passes through automated pre-screening. This reduces the manual review workload by flagging obvious issues automatically.
• Profanity and policy violation detection: run pack titles, descriptions, and all prompt bodies through a moderation API (OpenAI Moderation API - free, fast, accurate). Flag any content scoring above 0.7 on hate, sexual, violence, or self-harm categories. Flagged packs go to a "Needs Review" sub-queue in admin with the specific violation scores shown.
• Duplicate detection: hash each prompt body (SHA-256 normalised text) and check against existing published prompts. Flag packs with > 20% duplicate prompts. This prevents creators from re-selling platform-generated content.
• Quality scoring: a lightweight AI quality assessment runs on each prompt body and rates it on: specificity (is it a real actionable prompt or too vague?), length (too short = likely low quality), and structure (does it have clear instructions?). A quality score (0-100) is computed per prompt and averaged for the pack. Packs scoring below 40 are automatically rejected with a detailed improvement report sent to the creator.
12.2 Trust Score System  [Backend + Component]
Every user and creator has a Trust Score (0-100) computed from multiple signals. The score is internal (not shown to users directly) but affects: purchase limits for new accounts, review eligibility, appeal priority, and creator application acceptance.

13. Accessibility & Internationalisation
13.1 WCAG 2.1 AA Full Compliance
The current build has partial accessibility coverage. Full WCAG 2.1 AA compliance is required for a production-grade platform, both for legal compliance (ADA, EAA) and to reach the broadest possible audience.
• Screen reader support: all interactive elements have descriptive aria-labels. Dynamic content updates use aria-live regions (for toasts, cart updates, search results). Modal dialogs use role="dialog" with aria-labelledby and aria-describedby.
• Keyboard navigation: full keyboard operability throughout. Custom focus trap in modals (focus-trap-react). Skip-to-main-content link at the top of every page. Logical tab order on all forms.
• Colour contrast: all text meets 4.5:1 ratio (normal text) and 3:1 (large text). The current text-secondary colour (#A0A0C0 on #0D0D1A) achieves 7.8:1 - compliant. Verify all badge and button text similarly.
• Reduced motion: the existing prefers-reduced-motion implementation must be audited to ensure all animations (including the homepage hero, marquee, and carousel) are fully disabled or replaced with static states.
• Text resize: layout must not break at 200% browser text zoom. Test and fix any overflow or overlap at this zoom level.
• Error identification: all form errors identified by more than colour alone (icon + text, not just red border).
ℹ  Run automated audit: npx axe-cli https://your-domain.com --tags wcag2a,wcag2aa --exit - zero violations target.
13.2 Internationalisation (i18n)  [SCR-073 - System]
Phase 1 of i18n (Spanish and Japanese) represents a 2× addressable market expansion at relatively low implementation cost since the content itself (prompts) is already in English. The UI shell is what needs translating.
• Library: react-i18next with i18next-http-backend. Translation files stored in /public/locales/{lang}/translation.json.
• Language detection: browser language header (Accept-Language) on first visit. User preference stored in users.preferences.language. Language toggle in the footer and account settings.
• RTL support: while Spanish and Japanese are LTR, the system should be architected for RTL from the start using CSS logical properties (margin-inline-start instead of margin-left) to enable Arabic/Hebrew in the future.
• Content translation: pack titles, descriptions, and prompts themselves remain in their original language (English for platform-generated content). Creator-submitted packs can be in any language, indicated by a language badge on pack cards.
• Currency localisation: while all prices are stored in USD cents, display in the user's local currency (fetched via exchange rate API, cached daily in Redis) with appropriate formatting (¥1,200, €11.50, £9.99). Note: all transactions are processed in USD - the local currency display is cosmetic only.

These features are blocking user retention right now and should be started in Sprint 1 regardless of sprint planning. A user who creates an account, makes a purchase, and then finds they have no meaningful profile, no way to interact with prompts beyond downloading, and no way to browse or copy individual prompts will not return.
• Public user profile page (SCR-033) - identity and trust foundation
• Full prompt viewer with copy functionality (SCR-052) - core product value after purchase
• Notification centre (SCR-050) - the first touchpoint that brings users back
• Cart system (SCR-063) - enables multi-item purchasing and AOV growth
• Try-before-you-buy upgrade (SCR-064) - highest-impact conversion improvement
• Feature flags in admin (SCR-079) - enables safe staged rollout of all new features
20.2 Technical Dependencies
Several features have hard technical prerequisites that should be set up before their sprint begins. pgvector extension on PostgreSQL must be enabled before Sprint 4's semantic search work begins - this requires a database migration and cannot be added retroactively without downtime. Stripe Connect must be configured before Sprint 6's creator payout system - this requires Stripe account verification and platform settings that take 1-3 business days. VAPID keys for Web Push must be generated and stored in environment variables before Sprint 9's PWA push notifications work begins.
